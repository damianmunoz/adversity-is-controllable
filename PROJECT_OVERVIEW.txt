--------------------------------------------------------------------------------
  WHAT THIS PROJECT IS (AND IS NOT)
--------------------------------------------------------------------------------

This is NOT a price prediction system. We are not trying to guess where Bitcoin
will go. We are building something more interesting: an adaptive execution
engine that learns HOW to act in a market, not WHERE prices will move.

The core question is: given what we can observe about the market right now
(order flow, book shape, volatility), what is the smartest action to take
in order to minimize execution cost and decision error over time?

The three possible actions we are deciding between:
  - WAIT       → do nothing, let the market evolve
  - PASSIVE    → post a limit order (sit on the spread, may not fill)
  - AGGRESSIVE → hit the market (fill immediately, pay the spread)

To answer that question we combine two classical systems:
  1. Kalman Filter   — estimates hidden market state (pressure + regime)
  2. Hedge Algorithm — regret-minimizing policy that picks the best action

Right now, Steps 1–3 are complete. We have data flowing, books reconstructed,
and features computed. Steps 4–7 (state estimation, policy, simulation) are
next.

  THE CODEBASE — FILE BY FILE
--------------------------------------------------------------------------------

Every file lives under src/ or scripts/. The principle is: scripts are thin
entry points, all real logic lives in src/.

════════════════════════════════════════════════════════════════════════════════
  STEP 1 — INGEST (getting data from Binance)
════════════════════════════════════════════════════════════════════════════════

  src/ingest/ws_ingest.py
  -----------------------
  What it does:
    Opens a persistent WebSocket connection to Binance and listens to two live
    streams simultaneously: every trade that executes on BTC/USDT, and every
    change to the order book depth (at 100ms resolution). Each event is
    cleaned up (normalized) and written to disk.

  How it does it:
    Uses Python's asyncio + websockets library to maintain a long-running
    connection. If the connection drops, it automatically reconnects after 5
    seconds. Events arrive as JSON and are written one line at a time to .jsonl
    files, organized by hour (hour=04/events.jsonl, hour=05/events.jsonl, etc.)

  Why it matters:
    This is our data source. Without this running, there is no signal. The two
    streams we subscribe to are:
      - btcusdt@trade      → every single trade: price, quantity, direction
      - btcusdt@depth@100ms → which price levels in the book changed and by how much

  Key output fields (trades):
    event_time_ms, price, qty, is_buyer_maker (true = seller hit the bid)

  Key output fields (depth updates):
    first_update_id, final_update_id (for continuity checking), bids[], asks[]
    Each bid/ask entry is [price, new_quantity] — if quantity is 0, that level
    was removed from the book.

  ---

  src/ingest/rest_snapshot.py
  ---------------------------
  What it does:
    Takes a one-time HTTP snapshot of the full order book state from Binance's
    REST API. This is needed to initialize the book before we can replay
    incremental updates from the WebSocket.

  How it does it:
    Calls the Binance REST depth endpoint and saves the result as a JSON file,
    timestamped in the filename, under data/raw/depth_snapshots/.

  Why it matters:
    The WebSocket depth stream is a differential feed — it only tells you what
    changed, not the full state. You cannot reconstruct the book from diffs alone
    without knowing the starting state. This snapshot is the "seed."

════════════════════════════════════════════════════════════════════════════════
  STEP 2 — ORDER BOOK (reconstructing the live book from raw events)
════════════════════════════════════════════════════════════════════════════════

  src/book/order_book.py
  ----------------------
  What it does:
    An in-memory limit order book (LOB). It holds two sorted lists of price
    levels — all bids (buy orders) sorted highest-first, and all asks (sell
    orders) sorted lowest-first — and keeps them up to date as new events arrive.

  How it does it:
    Uses a SortedDict (sorted dictionary) for both sides. Bids are stored with
    negated price keys so the best (highest) bid is always at index 0. This
    gives O(log n) insert/delete at any price level and O(1) access to best bid
    and best ask — which is critical because Binance updates arrive at arbitrary
    price levels hundreds of times per second.

    When a depth update arrives, for each price level:
      - If new quantity > 0: update or add that price level
      - If new quantity = 0: remove that price level (it was fully consumed)

    It also checks for sequence gaps: Binance assigns monotonically increasing
    update IDs to every event. If an event arrives with a gap in those IDs, it
    means we missed events — the book may be stale. This is flagged.

  Why it matters:
    This is the central data structure of the whole project. Everything downstream
    depends on having an accurate, up-to-date picture of the order book. Without
    this, our features are meaningless.

  Key output type: BookSnapshot
    A compact read-only snapshot of the top-20 price levels on each side at a
    given moment. This is what gets passed to the feature pipeline.

  ---

  src/book/book_builder.py
  ------------------------
  What it does:
    Orchestrates the replay process. Seeds the OrderBook from a REST snapshot,
    then feeds it incremental depth update events one by one in chronological
    order, and emits a BookSnapshot every 10 events (~1 second).

  How it does it:
    Reads depth update events from JSONL files on disk in time order. For each
    event, applies it to the OrderBook. Every 10 events, takes a snapshot and
    yields it downstream. If a sequence gap is detected, it emits a snapshot
    immediately and flags it so downstream code can decide whether to skip it.

  Why it matters:
    This is the "playback engine" — it turns raw stored events back into a
    simulated live book. Without it, our JSONL files are just JSON sitting on
    disk. With it, we can replay market microstructure as if we were watching
    it happen in real time.

════════════════════════════════════════════════════════════════════════════════
  STEP 3 — FEATURES (extracting signal from the book)
════════════════════════════════════════════════════════════════════════════════

  This is the core signal extraction layer. We take each BookSnapshot and
  compute a set of numbers that characterize the current state of the market.
  These numbers are the inputs to the Kalman filter in Step 4.

  ---

  src/features/spread.py
  ----------------------
  What it does:
    Computes price and spread information from the top of the book.

  Outputs:
    - best_bid_px / best_ask_px  — the current best prices to buy and sell at
    - spread_abs                 — the gap between them in USDT (e.g., $1.20)
    - spread_bps                 — that same gap as basis points of mid price
                                   (e.g., 1.8 bps means very tight, liquid market)
    - mid_price                  — simple average of best bid and best ask
    - microprice                 — quantity-weighted mid price

  The microprice is the interesting one. If the best bid has 0.1 BTC sitting on
  it and the best ask has 10 BTC, the bid is much thinner and more likely to be
  consumed first — meaning price is more likely to fall. Microprice accounts for
  this by weighting the midpoint toward the thinner side. It is a better short-
  term price estimate than simple mid.

  ---

  src/features/ofi.py
  -------------------
  What it does:
    Computes Order Flow Imbalance (OFI) — a measure of the net pressure at each
    price level by comparing two consecutive snapshots.

  How it works:
    For each of the top 5 price levels:
      OFI_i = (how much bid quantity changed at level i)
            - (how much ask quantity changed at level i)

    If OFI is positive, buy pressure increased or sell pressure decreased.
    If OFI is negative, the opposite.

  Why it matters:
    OFI is one of the strongest short-term predictors of price movement in
    market microstructure research (Bouchaud et al.). It captures the flow of
    orders hitting the book, which tends to lead price changes by fractions of
    a second. We compute it across 5 levels (L1 through L5) rather than just
    the top because deeper levels can signal institutional accumulation before
    it shows up at L1.

  ---

  src/features/imbalance.py
  -------------------------
  What it does:
    Computes a single number in the range (-1, +1) summarizing the shape of
    the entire top-10 book in one shot.

  Formula:
    imbalance = (total bid quantity - total ask quantity)
              / (total bid quantity + total ask quantity)

    +1.0 = all liquidity is on the buy side (strong upward pressure)
    -1.0 = all liquidity is on the sell side (strong downward pressure)
     0.0 = perfectly balanced

  Volume weighting:
    Levels far from the mid price are less likely to trade and should count
    less. We apply a linear decay: level 1 counts the most, level 10 the least.

  Why it matters:
    Gives the Kalman filter a normalized, single-number view of directional
    pressure across the whole visible book. Unlike OFI (which is a delta),
    depth imbalance is a point-in-time measure.

  ---

  src/features/volatility.py
  --------------------------
  What it does:
    Maintains a rolling buffer of mid prices and computes realized volatility
    over three time windows: 1 second, 5 seconds, and 30 seconds.

  How it works:
    At each tick, computes log(current_mid / previous_mid) and adds it to a
    time-indexed buffer. Volatility for each window = standard deviation of log
    returns within that window. If fewer than 3 samples are in a window, returns
    0.0 (window not yet populated).

  Why three windows?
    - vol_1s  → captures microstructure noise and sudden spikes
    - vol_5s  → medium-term regime signal
    - vol_30s → slow trend / regime context

    Together they let the Kalman filter distinguish:
      "Calm" (all low) vs "Spike" (high 1s, low 30s) vs "Trending" (all rising)

  ---

  src/features/feature_pipeline.py
  ---------------------------------
  What it does:
    The orchestrator. Takes a stream of BookSnapshots and calls each of the
    four feature modules above, assembling the results into a single flat
    FeatureVector. Writes them to Parquet files on disk.

  How it works:
    It is stateful: it holds a reference to the previous snapshot (needed for
    OFI deltas) and maintains the RollingVolatility buffer across ticks. This
    means it must be created once and fed snapshots in order — not reset between
    ticks.

  Output: FeatureVector (one row per ~1 second tick)
    ts_ms, symbol, best_bid_px, best_ask_px, spread_abs, spread_bps,
    mid_price, microprice, ofi_l1..l5, depth_imbalance, vol_1s, vol_5s,
    vol_30s, sequence_gap

  Storage format: Apache Parquet
    Column-oriented, compressed, schema-enforced. PyArrow validates types at
    write time so a badly typed value raises an error immediately rather than
    silently corrupting the file.

--------------------------------------------------------------------------------
  THE PIPELINE AT A GLANCE — HOW EVERYTHING CONNECTS
--------------------------------------------------------------------------------

  [ Binance WS ] → trades.jsonl + depth_updates.jsonl   (Step 1: ingest)
       ↓
  [ OrderBook ]  → BookSnapshot every ~1 second          (Step 2: book)
       ↓
  [ Features ]   → FeatureVector (17 numbers)            (Step 3: features)
       ↓
  [ Kalman ]     → KalmanState [market_pressure, regime] (Step 4: state — NEXT)
       ↓
  [ Hedge ]      → Action: WAIT / PASSIVE / AGGRESSIVE   (Step 5: policy — NEXT)
       ↓
  [ Simulator ]  → FillResult → scalar loss              (Step 6: execution — NEXT)
       ↓
  [ Updater ]    → policy weights updated                (Step 7: loop — NEXT)

The first three steps convert messy real-time market data into clean, structured
numbers. Steps 4–7 are the intelligence layer — they take those numbers and
make decisions that get better over time through regret minimization.

════════════════════════════════════════════════════════════════════════════════
  STEP 4 — Kalman Filter  (src/state/kalman_filter.py)
════════════════════════════════════════════════════════════════════════════════

What it will do:
  Estimate two hidden quantities at every tick:
    x_hat[0] = market_pressure (net directional force — how hard the market
                                is pushing in one direction right now)
    x_hat[1] = regime          (volatility regime — calm vs turbulent)

  These are not directly observable. The Kalman filter infers them from what
  IS observable — our FeatureVector. Think of it like a GPS: you can't see
  your exact position, but you can estimate it from satellite signals with
  known noise. Kalman is optimal at this under Gaussian noise assumptions.

How it will work:
  Standard linear Kalman filter in 2D state space.
    - State transition: F matrix (how state evolves from tick to tick)
    - Observation: H matrix (which features best reflect the hidden state)
    - Process noise: Q (how much the state can jump between ticks)
    - Observation noise: R (how noisy our feature measurements are)

  Observations fed in: ofi_l1 through ofi_l5, depth_imbalance, microprice
  (these are the features most theoretically connected to market pressure)

  Output per tick: KalmanState(ts_ms, x_hat[2], P[2x2], innovation)
    P is the 2x2 uncertainty covariance — tells us how confident the filter is.
    innovation is the surprise term: how far the observation was from predicted.

Key design decisions to make:
  - What to use as the observation vector (subset of FeatureVector)
  - Initial values for F, H, Q, R (will need tuning against real data)
  - Whether to normalize features before feeding to the filter

Files to create:
  src/state/kalman_filter.py
  src/state/__init__.py
  configs/kalman.yaml
  scripts/4_run_kalman.py

════════════════════════════════════════════════════════════════════════════════
  STEP 5 — Hedge Policy  (src/policy/)
════════════════════════════════════════════════════════════════════════════════

What it will do:
  Given the current market state (KalmanState + FeatureVector), select one
  of three actions: WAIT, PASSIVE, AGGRESSIVE.

  The Hedge algorithm (also called Multiplicative Weights / Exponential
  Weights) is a classical online learning algorithm with provable regret
  bounds. It does not require a model of the market — it simply tracks which
  actions have been working and gives them more weight.

How it will work:
  Maintain a weight vector w = [w_WAIT, w_PASSIVE, w_AGGRESSIVE].
  Initially uniform (0.33, 0.33, 0.33).

  Select action: proportional to weights (randomized selection).

  After each action, observe the loss (computed in Step 6) and update:
    w(a) ← w(a) * exp(-η * L(a))

  where η (eta) is the learning rate and L(a) is the realized loss for
  that action. Actions with high loss get exponentially down-weighted.
  Over time the distribution concentrates on the best action.

Files to create:
  src/policy/actions.py    (Action enum: WAIT / PASSIVE / AGGRESSIVE)
  src/policy/hedge.py      (weight vector, select(), update(), save())
  src/policy/__init__.py
  configs/policy.yaml      (eta, initial weights)
  scripts/5_run_policy.py

════════════════════════════════════════════════════════════════════════════════
  STEPS 6+7 — Simulation + Loss + Outer Loop  (src/execution/)
════════════════════════════════════════════════════════════════════════════════

What it will do:
  Given an action from the policy, simulate what would have happened if we
  executed it against the real order book. Compute a loss (how bad was the
  outcome). Feed that loss back to the policy for its weight update.
  Repeat for every tick in the dataset.

  This is the closing of the loop. It turns the Kalman + Hedge stack into
  an adaptive system that learns from its own simulated decisions.

Execution logic:
  WAIT:       No fill. Loss = price move away from us (opportunity cost if
              the market moves against our eventual direction).
  PASSIVE:    Simulate a limit order at the best bid/ask. Fills if the market
              crosses that price during the next window; does not fill otherwise.
              If filled: loss = slippage (how far from mid) + adverse move.
  AGGRESSIVE: Simulate a market order. Always fills immediately at best ask
              (buy) or best bid (sell). Loss = spread paid + adverse move.

Loss formula:
  L = slippage + λ * adverse_move
  where:
    slippage     = |fill_price - arrival_mid|
    adverse_move = |exit_mid - arrival_mid| * direction_penalty
    λ            = risk aversion parameter (how much we penalize adverse moves
                   relative to slippage)

  src/execution/updater.py     (outer loop: all steps 4–7 together)

  LATER — Analysis and Evaluation
════════════════════════════════════════════════════════════════════════════════

After the full loop is running, the interesting questions become:

  1. Does the policy converge?
     Watch the weight vector over time. Does it stop being uniform and start
     concentrating on one or two actions? Does the distribution change across
     market regimes (calm vs volatile)?

  2. Does Kalman state track real regimes?
     Plot x_hat[1] (regime estimate) against realized volatility from the
     features. Do they correlate?

  3. Does the policy agree with market outcomes?
     Periods where AGGRESSIVE is preferred should correlate with high OFI
     and directional imbalance. WAIT periods should cluster around low-vol,
     mean-reverting regimes.

  4. Robustness check:
     Does the system behave well during known events in our 16-day window?
     What happened during any high-volatility periods? Did the Kalman filter
     detect the regime shift? Did the policy react?

  5. Extend to other instruments:
     The entire pipeline is symbol-agnostic. ETH/USDT, SOL/USDT — same code,
     different config file.
